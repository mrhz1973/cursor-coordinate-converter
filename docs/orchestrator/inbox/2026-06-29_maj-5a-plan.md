## 2. Obiettivo MAJOR-5A proposto

### Prodotto finale minimo (MVP programma)

Un **GIS Object Workbench** — pannello floating map-first che offre **una sola lista operativa** degli oggetti già creati dall'operatore, con:

- **Vista unificata** (tabella) di waypoint, tracce salvate e poligoni GIS (fase 1–2).
- **Filtro per tipo** + ricerca testuale sul nome.
- **Click riga → centra/evidenzia** sulla mappa (fly-to / highlight sessione).
- **Click mappa → seleziona riga** (fase 2+, solo quando sicuro vs modalità draw/edit).
- **Azioni leggere** che **delegano** ai pannelli/CRUD esistenti (Apri, ⌖ Centra, Rinomina dove già supportato, Elimina con conferma esistente, Copia coordinate).
- **Nessun nuovo store persistito**, nessun import/export unificato, nessun mission package.

### Oggetti inclusi (per fasi)

| Fase | Inclusi | Motivo |
|------|---------|--------|
| **5A1 MVP** | Waypoint, tracce **salvate**, poligoni GIS | Store maturi, liste/tabelle già ricche, fly-to esistente |
| **5A2** | + highlight mappa↔lista; pick mappa (waypoint → poligono → traccia) | Valore tattico selezione incrociata |
| **5A3** | + azioni Apri pannello nativo, delete/rename delegati, vis toggle dove esiste | Operatività senza duplicare editor |
| **Opzionale 5A4** | Range rings come righe read-only + fly | Utili come riferimento spaziale; non unificare delete con WP |
| **Backlog workbench** | `gisTracks[]` nativi | Scaffold senza UI/overlay — includere solo quando esiste percorso render |
| **Esclusi** | Preferiti, cronologia, batch, note, aree offline, misura draft, `state.track` draft | Semantica diversa o transiente |

### Perché è utile all'operatore

Oggi l'operatore deve aprire **tre pannelli separati** (Waypoint modal, Track modal/archivio, Polygon panel) per orientarsi sugli oggetti sulla mappa. In scenario tattico serve **orientamento rapido**: «cosa c'è qui?», «portami lì», «apri l'editor giusto» — senza rifare export/import o mission package.

   - Waypoint modal: tabella con edit/zoom/fav/delete (`renderWpModalList` ~52954).

### Pattern generici (senza attribuzione app esterna)

- **Catalogo oggetti + fly-to** — standard GIS leggero.
- **Selezione singola sessione** — evidenza temporanea, non layer manager enterprise.
- **Delega azioni** — workbench come **hub navigazione**, non secondo editor geometrico.

## 5. Design funzionale consigliato

### Lista oggetti unificata

**Adapter read-only** (nuova regione JS):

```text
workbenchCollectRows(opts) → [
  { kind, id, name, typeLabel, summary, when, visible, lat, lon, flyFn, openFn }
]
```

- **kind:** `waypoint` | `savedTrack` | `polygon` | (future `rangeRing`)
- **summary:** coordinate DD brevi, vertici/distanza, area — riuso helper esistenti (`polygonFeatureListMetrics`, `formatSavedTrackWhen`, ecc.)
- Ordinamento default: **tipo → nome → updated desc**; opzionale sort colonne come saved tracks (fase 2).

### Filtro per tipo

Chip/filtri: **Tutti | Waypoint | Tracce | Poligoni** — filtro client-side su array adapter, nessuna query rete.

### Ricerca testo

Input singolo: match case-insensitive su `name` + `summary` (debounce leggero opzionale, no persistenza).

### Click lista → mappa

- **Primario:** ⌖ / click riga → chiama fly esistente per kind:
  - waypoint: pan/zoom a lat/lon (pattern `wp-zoom`)
  - savedTrack: `flyToSavedTrackById`
  - polygon: bbox centroide o handler `data-role="poly-show"` esistente
- **Highlight sessione:** flag `state._workbench.highlight = { kind, id }` → overlay aggiunge stroke/marker ring **solo sessione** (5A2).

### Click mappa → lista (fase 2+)

- **Modalità pick workbench** esplicita (toggle nel pannello), mutuamente esclusiva con:
  - `waypointPickMode`, `track.pickMode`, polygon draw/edit, range pick, convert pick, measure
- **Ordine implementazione sicuro:** waypoint (marker hit) → poligono (bbox/proximity) → traccia (polyline proximity)
- Overlays attuali sono `pointer-events:none` → serve **hit-test** dedicato in pick mode (non cambiare default overlay).

### Azioni leggere (delega)

| Azione | Implementazione |
|--------|-----------------|
| **Centra** | fly helpers esistenti |
| **Apri pannello** | `openWaypointModal`, `openTrackModal`, `openPolygonPanel` + scroll/highlight riga nativa |
| **Rinomina** | apri pannello nativo o inline rename **solo** dove già esiste |
| **Elimina** | `waypointRemove`, saved track delete flow, `gisFeatureDelete` / polygon delete confirm — **mai** nuovo path distruttivo |
| **Mostra/Nascondi** | solo dove esiste (`savedTrackMapVisible`, `polygonMapVisible`) |
| **Copia coordinate** | clipboard testo DD/MGRS da lat/lon — read-only, no persist |

### Notifiche

Area interna al pannello (`role="status"`, `aria-live="polite"`) per errori pick, oggetto non trovato, conflitto tool attivo — standard GIS §5 notifiche interne.

## 6. Stato e dati

### Session-only (default)

```javascript
state._workbench = {
  open: false,           // o gisWorkbenchPanelOpen coerente con altri *PanelOpen
  filterKind: "all",     // all|waypoint|savedTrack|polygon
  search: "",
  selected: null,        // { kind, id } | null
  highlight: null,       // { kind, id } | null

```

**Non persistere** in `saveStore` / `coordconv_v2`.

### Cosa NON persistere

- Filtri, ricerca, selezione, highlight, pick mode, sort colonna workbench (salvo pattern sessione su nodo mount come poligon panel — accettabile **transiente**).

### Campo persistito — segnalazione

**Non proposto come default.** Eventuale futuro `properties.workbenchTag` su feature GIS = **decisione esplicita** team + sanitizer — **fuori MAJOR-5A**.

### Preservare `state.mapWaypoints[]`

- Workbench **legge** e **delega**; **non** duplica array.
- Nessuna migrazione verso `gisTracks` per waypoint.
- Export/import resta nei pannelli nativi.

### Evitare conflitti store

- **Non** unificare `savedTracks` e `gisTracks` in MVP.
- **Non** scrivere in `gisSelection[]` senza `{kind,id}` — preferire `_workbench`.
- Usare guard `workbenchCanActivate()` che verifica assenza di: polygon draw/edit, track pick, waypoint pick, measure attivo, range pick.

## 7. UI/UX — dove mettere il workbench

### Opzione A — **Pannello floating map-first** (consigliata)

- Nuovo `<dialog id="gisWorkbenchPanel">` o `#objectWorkbenchPanel` — stesso pattern `#polygonPanel`, `#measurePanel`, `#rangeRingsPanel`.
- Apertura da **pulsante toolbar mappa** (accanto a waypoint/track/polygon/measure).
- **Pro:** map-first, coerente con track/waypoint/polygon; resize e/w già standardizzato; Esc/chiusura già wired globalmente.
- **Contro:** +1 pulsante toolbar (mobile: verificare hit target 36px già usato in BUNDLE-E).

### Opzione B — Tab `#appTopbar`

- **Contro:** topbar già limitata (favorites, geocoding, history, layers); drawer laterale meno map-centric; waypoints/track già esclusi da tab drawer.
- **Pro:** scopribilità menu.
- **Verdetto:** **non consigliata** come primaria.

### Opzione C — Strumenti (`#toolsModal` / `GIS_TOOL_SECTIONS`)

- Pattern Diagnostica (`sec-diagnostics`).
- **Pro:** zero toolbar clutter.
- **Contro:** meno map-first; operatore deve aprire Strumenti → secondario.
- **Verdetto:** entry **secondaria** opzionale (link «Apri workbench»), non primaria.

### Opzione D — Estendere tab Layers

- **Contro:** Layers = offline tile; mescola semantica infrastruttura vs oggetti tattici.
- **Verdetto:** **rifiutata**.

### Layout tabella

- Riutilizzare classi `.saved-tracks-table` / `.poly-list-table`: sticky header, scroll interno, colonne min-width, resize opzionale fase 2.
- Colonne MVP: **Tipo | Nome | Riepilogo | Quando | Vis | Azioni** (⌖ Apri …).

### Mobile

- Pannello floating full-height fraction (~0.75) come Search B1; scroll orizzontale tabella se necessario.

## 10. Rischi e non-goals

### Fuori scope (confermato)

- MAJOR-2E, MAJOR-3 import/export unificato, MAJOR-4 mission package
- OPSEC, fetch, proxy, geocoding, tile, cache, IndexedDB
- Live tracking GPS / GPS silenzioso
- Schema dati grosso, nuovi campi persistiti, migrazione store
- Refactor architetturale (`state` centralizzato, event delegation globale)
- Micro-polish non funzionale, titolo `<title>`, i18n FR nuove (preferire IT/EN + riuso chiavi)
- Unificare `savedTracks` ↔ `gisTracks` ↔ `state.track`
- Undo/redo globale (`undoStack` resta scaffold)

### Rischi principali

1. **Conflitto modalità mappa** — pick/draw/edit simultanei → guardie obbligatorie + feedback in-pannello.
2. **Duplicazione UI** — workbench deve **delegare**, non sostituire editor poligono/traccia.
3. **`gisTracks[]` vuoto/invisibile** — includere troppo presto confonde; rimandare.
4. **Overlay pointer-events:none** — pick mappa richiede layer hit-test dedicato (5A2).
5. **Governance i18n FR** — nuove stringhe **IT/EN**; riusare chiavi `track.colName`, `tip.trackSavedFly`, ecc.
